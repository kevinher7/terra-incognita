# Camera Trap Failure Mining Dashboard — Design

> The dashboard (consumer) design. Cross-repo contracts with the training repo
> (`terra-incognita`) and the infra repo are **not restated here** — they live in
> [`../contracts/`](../contracts/) and are linked inline. To change a contract,
> change it there, never here.

**Status:** Ready for implementation planning
**Owner:** Kevin Hernandez
**Date:** 2026-05-31

---

## 1. Goal

Portfolio project targeting **ML platform/tooling engineer** roles. Demonstrates
building diagnostic tooling that helps engineers understand model behavior — not
just training a model, but the infrastructure around it.

**Resume framing:**
> Developed a computer vision failure-analysis dashboard for object detection
> models, automatically categorizing prediction errors and surfacing dataset
> gaps for targeted data collection.

---

## 2. Scope

### MVP
- Single pre-registered dataset, batch inference via MLflow serving
- Error categorization (6 types) with hardcoded IoU/confidence thresholds
- Dashboard: worst failures, error distribution, per-image detail view
- Filters by class, confidence, error type
- Model/dataset selection from MLflow registry (sync on startup + refresh)
- WebSocket progress for inference/categorization runs
- JWT auth with single shared password
- Docker-based deploy-on-merge CI/CD

### Post-MVP
- Visual similarity clustering (CLIP embeddings + FAISS)
- Suggested next actions ("collect more night images", "species A confused with B")
- Extra filters: brightness, time of day, camera/location
- Sequence-based failure grouping (burst analysis)

### Explicitly Excluded
- Video support
- Auto-generated reports
- Training pipeline (separate repo with MLflow)
- Infrastructure code (separate repo with Terraform)

## 3. Architecture

```
Browser (Vite + React + TanStack Router/Query + shadcn/ui + Recharts)
    |
    |-- fetches data --> FastAPI (Python, on EC2)
    |                        |-- reads/writes SQLite (EBS, backed up to S3)
    |                        |-- generates presigned S3 URLs for images
    |                        |-- discovers models (registry aliases) + datasets
    |                        |   (the "datasets" experiment) via MLflow
    |                        |-- calls MLflow serving for inference (base64 in)
    |                        |-- runs error categorization (in-process module)
    |                        |-- WebSocket for run progress
    |                        +-- JWT auth (single shared password)
    |
    +-- loads images --> S3 (via presigned URLs)

MLflow (separate infra, separate repo) — see ../contracts/mlflow-topology.md
    |-- model registry + serving (see ../contracts/model-registry.md, ../contracts/serving-io.md)
    +-- dataset tracking via the "datasets" experiment (see ../contracts/dataset-conventions.md)

CI/CD: GitHub Actions
    |-- PR: lint (ruff + eslint) + pytest
    +-- merge to main: build Docker -> push ECR -> deploy EC2
```

**Key separation:** The dashboard is a diagnostic tool, not a treatment tool.
It surfaces insights (failure modes, dataset gaps, class confusion patterns).
Engineers take those insights to the training pipeline to act on them. The
dashboard is a **pure consumer** of MLflow: it never triggers training and never
writes to the registry.

## 5. Data Model

| Entity | Key Fields | Notes |
|---|---|---|
| Dataset | id, name, mlflow_dataset_name, mlflow_dataset_version, s3_uri, coco_annotation_key, created_at | Discovered via the `datasets` experiment — see [../contracts/dataset-conventions.md](../contracts/dataset-conventions.md) |
| Image | id, dataset_id, file_name, s3_key, width, height, datetime, location_id, seq_id, frame_num | From COCO annotations |
| Category | id, name | **From COCO annotation file** (authoritative source of classes) |
| Location | location_id, gps_lat, gps_lon | One camera per location in CCT |
| Model | id, name, architecture, mlflow_model_name, mlflow_model_version, alias, created_at | Synced from MLflow registry; `alias` (e.g. `champion`) replaces the removed stage field — see [../contracts/model-registry.md](../contracts/model-registry.md) |
| Run | id, model_id, dataset_id, status, created_at, completed_at, config_json | IoU thresholds in config |
| GroundTruth | id, image_id, category_id, bbox_x1, bbox_y1, bbox_x2, bbox_y2 | From COCO annotations; stored `xyxy` — see [../contracts/bbox-format.md](../contracts/bbox-format.md) |
| Prediction | id, run_id, image_id, category_id, bbox_x1, bbox_y1, bbox_x2, bbox_y2, confidence | From inference; stored `xyxy`; join key is the COCO id returned by the pyfunc — see [../contracts/serving-io.md](../contracts/serving-io.md) |
| ErrorClassification | id, prediction_id, ground_truth_id, error_type, iou_value, confused_with_category_id | One of 6 types |

**Bounding-box storage.** Both `GroundTruth` and `Prediction` store absolute-pixel
`[x1, y1, x2, y2]`, and IoU is computed in that format. The full rule (including
the two allowed conversion boundaries) is the
[bbox-format contract](../contracts/bbox-format.md).

**Source of truth for classes.** The `Category` table is built by parsing the
COCO annotation file downloaded from S3; the training repo's `class_map_json` tag
is a convenience mirror only. See the
[dataset-conventions contract](../contracts/dataset-conventions.md).

**Post-MVP:** `embedding BLOB` column on Image + separate FAISS `.index` file.

## 6. Error Categorization (6 Types)

All IoU is computed in the stored box format (see
[../contracts/bbox-format.md](../contracts/bbox-format.md)).

| Type | Definition | Threshold (hardcoded MVP) |
|---|---|---|
| False positive | Detection with no matching GT box | IoU < 0.5 with all GT |
| False negative | GT box with no matching detection | No pred with IoU >= 0.5 |
| Localization error | Correct class, poor IoU | 0.3 <= IoU < 0.5 |
| Class confusion | Right location, wrong class | IoU >= 0.5, class mismatch |
| Low-confidence correct | Right but unsure | Correct + confidence < 0.5 |
| High-confidence wrong | Confidently wrong | Wrong + confidence >= 0.8 |

## 7. Dashboard Pages (MVP)

- **Run management:** Start batch inference, view run history, WebSocket progress bar
- **Error overview:** Distribution of error types (Recharts bar/pie), per-class breakdown
- **Failure browser:** Grid of worst predictions, image thumbnails via presigned URLs
- **Filters:** Class, confidence range, error type
- **Model/dataset selector:** Dropdown from MLflow, sync on startup + refresh button
- **Image detail:** Click failure -> image with GT and prediction boxes overlaid, metadata

## 9. MLflow Integration Points

The dashboard consumes, never produces. Three integration points, each governed
by a cross-repo contract (do not restate them here):

- **9.1 Datasets** — discovered via the `datasets`-experiment convention.
  Contract: [../contracts/dataset-conventions.md](../contracts/dataset-conventions.md).
- **9.2 Models** — resolved via registry aliases (not the removed stage field).
  Contract: [../contracts/model-registry.md](../contracts/model-registry.md).
- **9.3 Inference** — the pyfunc serving wire format (base64 in; detections out).
  Contract: [../contracts/serving-io.md](../contracts/serving-io.md).

MLflow placement / backend store / serving topology:
[../contracts/mlflow-topology.md](../contracts/mlflow-topology.md).

Sync strategy for datasets + models: pull on startup + on-demand refresh button
in UI.

## 12. Decisions Log

| # | Decision | Rationale |
|---|---|---|
| 1 | Python backend (FastAPI) | ML ecosystem is Python; target role expects it |
| 2 | Backend-first API design | Simpler; FastAPI auto-generates OpenAPI |
| 3 | SQLite on EBS + S3 backup | Cheap; sufficient for portfolio; cron backup handles Terraform recreation |
| 4 | S3 presigned URLs | Private bucket, no public access |
| 5 | Error categorization in-process | Not complex enough for Lambda/microservice |
| 6 | Training in separate repo | Different lifecycle; MLflow bridges the gap |
| 7 | Infrastructure in separate repo | Different lifecycle; Terraform concerns isolated |
| 8 | Docker deployment | Reproducible; portfolio signal |
| 9 | WebSockets for run progress | Learning goal; scope contained to one feature |
| 10 | Hardcoded thresholds for MVP | Simplicity; configurable later |
| 11 | Stratified subset sampling | Rare classes appear; more interesting failures |
| 12 | Vite + React (not TanStack Start) | No SSR needed; static build -> S3 |
| 13 | Minimal B&W UI (shadcn/ui) | Target role values pipeline over polish |
| 14 | JWT with shared password | Minimal auth; no user management |
| 15 | MLflow for models AND datasets, via the `datasets`-experiment convention | Single external dependency; datasets discovered via `search_runs(experiment_names=["datasets"])` since MLflow has no queryable dataset registry — contract: [../contracts/dataset-conventions.md](../contracts/dataset-conventions.md) |
| 16 | Dashboard = diagnostic only | Surfaces insights; doesn't trigger training |
| 17 | Model promotion via registry **aliases**, not stages | Stages are removed in MLflow 3.x; alias is the source of truth — contract: [../contracts/model-registry.md](../contracts/model-registry.md) |
| 18 | Bounding boxes stored `xyxy` end-to-end | Matches YOLO output + IoU math; one format avoids per-comparison conversion bugs; GT converted once on ingest — contract: [../contracts/bbox-format.md](../contracts/bbox-format.md) |
| 19 | Pinned pyfunc serving contract | Stateless, S3-decoupled serving; pyfunc translates YOLO index → COCO id so predictions join the `Category` table directly — contract: [../contracts/serving-io.md](../contracts/serving-io.md) |
